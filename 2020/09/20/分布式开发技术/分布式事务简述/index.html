分布式事务简述

本地事务简述

事务指逻辑上的一组操作，组成这组操作的各个单元，要么全部成功，要么全部不成功。从而确保了数据的准确与安全。

事务的四大特性

  • 原子性（Atomicity）：原子性是指事务是一个不可分割的工作单位，事务中的操作要么都发生，要么都不发生。

  • 一致性（Consistency）：事务必须使数据库从一个一致性状态变换到另外一个一致性状态。 例如转账前A有1000，B有1000。转账后A+B也得是2000。

  • 隔离性（Isolation）：事务的隔离性是多个用户并发访问数据库时，数据库为每一个用户开启的事务，每个事务不能被其他事务的操作数 据所干扰，多个并发事务之间要相互隔离。

  • 持久性（Durability）：持久性是指一个事务一旦被提交，它对数据库中数据的改变就是永久性的，接下来即使数据库发生故障也不应该对 其有任何影响。

事务的四种隔离级别

Serializable：可避免脏读、不可重复读、虚读情况的发生。（串行化） 最高

Repeatable read：可避免脏读、不可重复读情况的发生。(有可能发生幻读) 第二

Read committed：可避免脏读情况发生。 第三

Read uncommitted：最低级别，以上情况均无法保证。(读未提交) 最低

注意：级别依次升高，效率依次降低

MySQL的默认隔离级别是：REPEATABLE READ。（Oracle的默认是：READ COMMITTED）

隔离级别导致的问题

脏读：一个线程中的事务读到了另外一个线程中未提交的数据。

不可重复读：一个线程中的事务读到了另外一个线程中已经提交的update的数据（前后内容不一样）

虚读（幻读）：一个线程中的事务读到了另外一个线程中已经提交的insert的数据（前后条数不一样）

分布式事务

概念

分布式事务是指事务的参与者、支持事务的服务器、资源服务器以及事务管理器分别位于不同的分布式系统的不同节点之上。简单来说就是组成事务的各个单元处于不同数据库服务器上。

分布式事务的应用架构

解决方案分类

  • 刚性事务

    刚性事务指的就是遵循本地事务四大特性（ACID)的强一致性事务。它的特点就是强一致性，要求组成事务的各个 单元马上提交或者马上回滚，没有时间弹性，要求以同步的方式执行。通常在单体架构项目中应用较多，一般都是企业级应用（或者局域网应用）。例如：生成合同时记录日志，付款成功后生成凭据等等。但是，在时下流行的互联网项目中，这种刚性事务来解决分布式事务就会有很多的弊端。其中最明显的或者说最致命的就是性能问题。如下图所示：

    因为某个参与者不能自己提交事务，必须等待所有参与者执行OK了之后，一起提交事务，那么事务锁住的时间就变得非常长，从而导致性能非常低下。基于此我们也就得出了一个结论，阶段越多，性能越差。

  • 柔性事务

    柔性事务是针对刚性事务而说的，刚性事务有两个特点，第一个强一致性，第二个近实时性（NRT）。而柔性事务的特点是不需要立刻马上执行（同步性），且不需要强一致性。它只要满足基本可用和最终一致就可以了。要想真正的明白，需要从BASE理论和CAP理论说起。

CAP理论和BASE理论

CAP、BASE跟TCC补偿型方案有直接的关系，这两个分布式的理论对我们研究分布式系统里面的一些技术和方案都是作为基础的知识需要掌握的。

CAP理论

其中C、A、P的说明如下：

  • 一致性(Consistency)：在同一时刻，所有节点是否能获取到相同的数据。

  • 可用性(Availability)：这个比较好理解，就是节点是否还能响应客户端的读写请求。

  • 分区容忍性(Partition tolerance)：就是集群内部出现了分区，比如脑裂情况，节点无法达成一致，分区节点各自为战，仍可以对外提供服务。但这时候必须在C跟A之间必须要进行一个选择。

CAP原则指的是，这三个要素最多只能同时实现两点，不可能三者兼顾。因此在进行分布式架构设计时，必须做出取舍。而对于分布式数据系统，分区容错性是基本要求，否则就失去了价值。因此设计分布式数据系统，就是在一致性和可用性之间取一个平衡。对于大多数web应用，其实并不需要强一致性，因此牺牲一致性而换取高可用性，是目前多数分布式数据库产品的方向。。

BASE理论

BASE理论是指，Basically Available（基本可用）、Soft-state（ 软状态/柔性事务）、Eventual Consistency（最终一致性）。

  • 基本可用 BA：（Basically Available ）：

    指分布式系统在出现故障的时候，允许损失部分可用性，保证核心可用，但不等价于不可用。比如：正常情况下所有节点共同承担查询请求，但于故障，降级为所有请求都去访问主节点。通知进行限流操作，对部分请求返回一个稍后再进行查询。这时虽然比正常情况差了很多，但是系统还是可用的。

  • 软状态 S：（ Soft State）：

    软状态是指允许数据存在中间状态，并且该中间状态不会影响数据的一致性。即允许数据在不同节点间副本同步的时候存在延时。简单来说就是状态可以在一段时间内不同步。

  • 最终一致性 E：（Eventually Consistent ）：

    系统中的所有数据副本经过一定时间后，最终能够达到一致的状态，不需要实时保证系统数据的强一致性。最终一致性是弱一致性的一种特殊情况。BASE理论面向的是大型高可用可扩展的分布式系统，通过牺牲强一致性来获得可用性。ACID是传统数据库常用的概念设计，追求强一致性模型。简单来说就是在一定的时间窗口内， 最终数据达成一致即可。

BASE理论是基于CAP原则演化而来，是对CAP中一致性和可用性权衡的结果。

核心思想：即使无法做到强一致性，但每个业务根据自身的特点，采用适当的方式来使系统达到最终一致性。

刚性事务和柔性事务的比较

事务类型 时间要求 一致性要求 应用类型 应用场景
刚性事务 立即 强一致性 企业级应用（单体架构） 订单/订单项/日志
柔性事务 有时间弹性 最终一致性 互联网应用（分布式架构） 订单/支付/库存

相关协议及标准

DTP标准（DTP模型）

DTP标准，也称为DTP模型。它的全称是：Distributed Transaction Processing，是X/Open提出的一个分布式事务行业标准。X/Open，即现在的open group，是一个独立的组织，主要由各大知名公司或者厂商进行支持，这些组织不光遵循X/Open组织定义的行业技术标准，也参与到标准的制定。下图展示了open group目前主要成员(官网截图)：

X/Open针对DTP提供了以下参考文档： Distributed Transaction Processing: Reference Model和Distributed Transaction Processing: The XA Specification。它里面包含了DTP模型和XA规范。 在DTP规范中，定义了DTP模型和XA规范。

DTP模型有5个基本元素组成，他们分别是：

名称 简称 说明
应用程序(Application Program) AP 负责事务边界(即定义事务的开始和结束)的定义，并且在事务边界内对资源进行操作。
资源管理器(Resource Manager) RM 如数据库、文件系统等，并提供访问资源的方式。
事务管理器(Transaction Manager ) TM 负责分配事务唯一标识，监控事务的执行进度，并负责事务的提交、回滚等。
通信资源管理器(Communication Resource Manager) CRM 控制一个TM域(TM domain)内或者跨TM域的分布式应用之间的通信。
通信协议(Communication Protocol) CP 提供CRM提供的分布式应用节点之间的底层通信服务。

分布式事务解决方案

二阶段提交方案（2PC）

二阶段提交方案是基于XA协议规范设计的一套分布式事务解决方案或者说是协议，Two-Phase-Commitment-Protocol。主要包括两个阶段：

  • 准备阶段（Prepare）：TM向所有RM发送prepare消息，执行SQL语句但是不提交
  • 提交阶段（Submit）：如果准备中某个RM执行时失败或者超时，则TM向所有RM发送回滚消息，进行回滚。如果所有RM都执行成功，则提交事务。

存在的不足：

prepare阶段开始后，会占有操作的资源，直到submit阶段执行完成。如果在submit阶段，TM宕机或者RM无法接收commit消息，会导致一些RM收不到commit消息。

三阶段提交方案（3PC）

三阶段提价方案主要是对二阶段提价方案的一些改进，解决二阶段提价方案的一些问题。3PC的过程：

  • CanCommit阶段：TM发送一个CanCommit消息到所有RM，判断RM当前状态是否正常。

  • PreCommit阶段：如果CanCommit阶段执行成功，则TM发送PreCommit消息给到所有RM，相当于2PC里的Prepare阶段，执行SQL语句，但并不提交。

  • DoCommit阶段：如果PreCommit阶段执行成功，则TM发送DoCommit消息到所有RM，提交事务。如果PreCommit阶段某个 RM返回失败，则发送回滚消息，进行事务回滚。

改进点：

  • 引入CanCommit阶段，进行RM状态的判断。
  • PreCommit阶段是建立在所有RM状态正常的基础上。
  • DoCommit阶段增加超时机制，如果RM长时间收不到DoCommit消息，并且自己本身的PreCommit阶段执行成功，在等待一段时间后就会执行DoCommit操作。

存在缺陷：

如果在PreCommit阶段执行出现问题，但是某个RM在PreCommit操作执行成功后，没有接收到回滚消息，会导致数据不一致问题。

最终一致性方案

本地消息表

这种实现方式应该是业界使用最多的，其核心思想是将分布式事务拆分成本地事务进行处理，这种思路是来源于ebay。大概的执行流程如下：

  1. A系统在自己本地一个事务里操作同时，插入一条数据到消息表
  2. 接着A系统将这个消息发送到MQ中去
  3. B系统接收到消息之后，在一个事务里，往自己本地消息表里插入一条数据，同时执行其他的业务操作，如果这个消息已经被处理过了，那么此时这个事务会回滚，这样保证不会重复处理消息
  4. B系统执行成功之后，就会更新自己本地消息表的状态以及A系统消息表的状态
  5. 如果B系统处理失败了，那么就不会更新消息表状态，那么此时A系统会定时扫描自己的消息表，如果有没处理的消息，会再次发送到MQ中去，让B再次处理
  6. 这个方案保证了最终一致性，哪怕B事务失败了，但是A会不断重发消息，直到B那边成功为止

它和MQ事务消息的实现思路都是一样的，都是利用MQ通知不同的服务实现事务的操作。不同的是，针对消息队列的信任情况，分成了两种不同的实现。本地消息表它是对消息队列的稳定性处于不信任的态度，认为消息可能会出现丢失，或者消息队列的运行网络会出现阻塞，于是在数据库中建立一张独立的表，用于存放事务执行的 状态，配合消息队列实现事务的控制。

它的缺点在于依赖数据库的消息表进行事务管理，消息表会耦合到业务系统中，如果没有封装好的解决方案，会有很多预料之外的情况需要处理，而且不太适用于高并发场景。

MQ事务消息

与基本一致，不过去掉了消息表，直接基于MQ进行事务管理。目前一些大公司的事务也是基于MQ事务消息实现的，

有一些第三方的MQ是支持事务消息的，比如RocketMQ，ActiveMQ，他们支持事务消息的方式也是类似于采用的二阶段提交。但是有一些常用的MQ也不支持事务消息，比如 RabbitMQ 和 Kafka 都不支持。

以阿里的 RocketMQ 中间件为例，其思路大致为：

第一阶段Prepared消息，会拿到消息的地址。

第二阶段执行本地事务。

第三阶段通过第一阶段拿到的地址去访问消息，并修改状态。

也就是说在业务方法内要想消息队列提交两次请求，一次发送消息和一次确认消息。如果确认消息发送失败了RocketMQ会定期扫描消息集群中的事务消息，这时候发现了Prepared消息，它会向消息发送者确认，所以生产方需要实现一个check接口，RocketMQ会根据发送端设置的策略来决定是回滚还是继续发送确认消息。这样就保证了消息发送与本地事务同时成功或同时失败。下图描述了它的工作原理：

最大努力通知方案

与上述两种方案的不同在于，最大努力通知方案限制了系统B的执行次数，这个方案的大致意思就是：

  1. 系统A本地事务执行完之后，发送个消息到MQ

  2. 这里会有个专门消费MQ的最大努力通知服务，这个服务会消费MQ然后写入数据库中记录下来，或者是放入个内存队列也可以，接着调用系统B的接口

  3. 要是系统B执行成功就ok了；要是系统B执行失败了，那么最大努力通知服务就定时尝试重新调用系统B，反复N次，最后还是不行就放弃。

TCC补偿型方案

通用型TCC方案

TCC是补偿型分布式事务解决方案，TCC分别指的是Try，Conﬁrm，Cancel三个阶段：

  • Try阶段主要是对业务系统做检测及资源预留。

  • Conﬁrm阶段主要是对业务系统做确认提交，Try阶段执行成功并开始执行Conﬁrm阶段时，默认Conﬁrm阶段是不会出错的。即：只要Try成功，Conﬁrm一定成功。

  • Cancel阶段主要是在业务执行错误，需要回滚的状态下执行的业务取消，预留资源释放。

由此，我们可以得出结论，就是在Try阶段进行尝试提交事务，当Try执行OK了，Conﬁrm执行，且默认认为它一定成功。但是当Try提交失败了，则由Cancel处理回滚和资源释放。

TCC流程图

异步确保型TCC技术方案

如果要接入到一个TCC分布式事务中来，从业务服务必须将自己的业务拆分为三个步骤，对应着try、cancel、commit三个接口，改造起来比较麻烦。

异步确保型TCC技术方案，就是在主业务服务和从业务服务之间加了一个可靠消息服务。主业务服务的try、confirm和canel操作都调用可靠消息服务，在try阶段插入一条消息到本地数据库，在confirm阶段操作可靠消息服务调用主业务的实际业务接口，如果调用失败，则主业务服务发起cancel，可靠消息服务删除自己本地的消息即可。

补偿性TCC解决方案

跟通用型的TCC方案类似，只不过从业务服务只需要提供两个接口：Do和Compensate，就是执行接口和补偿接口，这种方案的好处就是折中一下了，不需要从业务服务提供锁定资源的接口，只需要加一个补偿接口，如果业务逻辑执行失败之后，进行补偿。

TCC和2PC的优劣分析

TCC事务的处理流程与2PC两阶段提交做比较，首先TCC是柔性事务，只要符合最终一致性即可。而2PC是刚性事务，它是强一致性的，在任何一个分布式阶段没有返回执行成功或失败的结果时，其事务一直会处于等待状态。并且2PC是利用DTP模型和XA规范，要求数据库支持XA规范，且通常都是在跨库的DB层面。

而TCC则在应用层面的处理，需要通过自己编写逻辑代码来实现补偿。它的优势在于，可以让应用自己定义数据操作的粒度，使得降低锁冲突、提高吞吐量成为可能。而不足之处则在于对应用的侵入性非常强，业务逻辑的每个分支都需要实现try、conﬁrm、cancel三个操作。此外，其实现难度也比较大，需要按照网络状态、系统故障等不同的失败原因实现不同的回滚策略。

TCC方案的一些特殊情况

  • 空回滚：try阶段由于一些问题，并没有真正请求到try接口，这时会被判定为失败，然后直接调用cancel接口，这时必须保证过cancel接口不做任何操作。

  • try回滚以及confirm回滚：try阶段与confirm阶段的失败都会执行cancel接口，如果是try阶段回滚，必须回滚掉try阶段的操作，如果是confirm阶段回滚，就要取消掉confirm阶段的操作。

  • 倒置请求：如果空回滚之后，接收到了try接口的调用，这时必须表征try接口不会被执行。

  • 接口幂等：在执行三个接口的调用时，有可能会发生重复调用的情况，必须保证三个接口具有幂等性。

Saga长事务

Saga与TCC事务比较相似，Saga是将每个接口拆分为2个接口，一个是业务接口，另外一个是补偿接口，相当于将TCC里面的try和confirm合并为一个接口。如果在服务调用链条里面，某个服务的业务接口执行失败了，那么直接对已经执行成功的所有服务都调用其补偿接口，将之前执行成功的业务逻辑给回滚。

Saga主要是两种思想，一种是编排模式，一种是命令模式。编排模式是事件驱动类型的模式，每个服务执行成功了就发布事件，下一个服务会监听到这个事件，然后继续执行。优点在于去中心化，缺点在于如果连续调用的服务太多，就会导致服务对消息的监听非常复杂，导致不好调试和定位问题。命令模式是指某个服务如果要开启一个saga分布式事务，那么就得创建一个类似saga流程管理器的东西，来负责依次调用和执行各个服务，如果某个服务调用失败，就对之前调用成功的服务依次执行补偿接口

编排模式，是事件驱动类型的模式，就是每个服务执行成功了自己发布个事件，下一个服务会监听到这个事件，然后继续执行，类似这样子，画个图来解释一下好了，那谁知道要监听谁的事件呢？

各种方案的适用场景

  • XA（两阶段提交）：适用于单系统多数据库的分布式事务
  • TCC（补偿型的分布式事务）：特别适用于微服务场景，而且支持复杂调用链条的场景。对dubbo技术栈、spring cloud技术栈的支持比较好。
  • 可靠消息最终一致性方案：比较适用于不是微服务复杂调用链条的场景，适用于的是调用之间可以通过MQ异步解耦，走异步化的方式，在不太复杂的一个服务调用场景里可以使用，主要是将比较耗时的一些操作放在MQ异步化的方式来实现，同时还能实现最终一致性的一个事务的效果。
  • 最大努力通知方案：比较适用于哪怕是数据不一致也可以的非关键性的操作。与可靠消息方案的区别在于重试多次，实在执行不成功就算了。
  • saga事务：类似于TCC，saga事务也可以用于微服务复杂服务调用链条的场景。
