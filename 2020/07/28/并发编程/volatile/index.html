volatile

基本概念

Java 内存模型中的可见性、原子性和有序性：

  • 可见性，是指线程之间的可见性，一个线程修改的状态对另一个线程是可见的。

  • 原子性，指的是这个操作是原子不可拆分的，不允许别的线程中间插队操作。

  • 有序性指的是你写的代码的顺序要和最终执行的指令保持一致。因为在Java内存模型中，允许编译器和处理器对指令进行重排序，重排序过程不会影响到单线程程序的执行，却会影响到多线程并发执行的正确性。

volatile要解决的就是可见性和有序性问题。

JMM

Java内存模型分为主内存和线程工作内存两大类。

  • 主内存：多个线程共享的内存。方法区和堆属于主内存区域。
  • 线程工作内存：每个线程独享的内存。虚拟机栈、本地方法栈、程序计数器属于线程独享的工作内存。

Java内存模型与CPU的多级缓存结构类似，但java内存模型是标准化的，屏蔽掉了底层硬件上的差别。

java内存模型规定，所有变量都需要存储在主内存中，线程需要时，可以在自己的工作内存保存变量的副本，线程对变量的所有操作都在工作内存中进行，执行结束后再同步到主内存中去。这里必然会存在时间差，在这个时间差内，该线程对副本的操作，对于其他线程是不见的，从而造成了可见性问题。

内存屏障

共有四种内存屏障，具体实现与 CPU 架构相关，不必钻研太深，只需知道它们的效果

LoadLoad

  • 防止 Load2 重排到 Load1 之前

LoadStore

  • 防止 Store 被重排到 Load 之前

StoreStore

  • 防止 Store1 被重排到 Store2 之后

StoreLoad(*)

  • 防止 Store 被重排到 Load 之后

使用方式

  • LoadLoad + LoadStore = Acquire：确保第一个 Load 操作在之后的 Load、Store 操作之前，保证第一个 Load 操作获取到正确的值。
  • LoadStore + StoreStore = Release：确保后一个 Store 操作不会被重排 Load、Store 操作之前，保证后一个 Store 操作写入的值，不会对之前操作有影响。
  • StoreLoad：比较特殊，用在线程切换时，对变量的写操作 + 读操作做同步，只要是对同一变量先写后读，那么屏障就能生效。

实现原理

当对volatile变量进行写操作的时候，JVM会向CPU发送一条lock前缀的指令，将这个缓存中的变量回写到系统主存中。同时因为有缓存一致性协议（MESI）的存在，CPU会通过嗅探在总线上传播的数据来检查自己缓存的值是不是过期，一旦发现过期就会将当前处理器的缓存行设置成无效状态，强制从主内存读取，这就保障了可见性。

而且volatile变量，通过内存屏障可以禁止指令重排，从而实现指令的有序性：

  • 每个volatile写操作前面，加StoreStore屏障，禁止上面的普通写和他重排；

  • 每个volatile写操作后面，加StoreLoad屏障，禁止跟下面的volatile读/写重排；

  • 每个volatile读操作后面，加LoadLoad屏障，禁止下面的普通读和voaltile读重排；

  • 每个volatile读操作后面，加LoadStore屏障，禁止下面的普通写和volatile读重排。
